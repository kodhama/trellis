# 0020 — Goals & examples are baked into the rules; the landing page is a derived view

**Raised by:** the maintainer — *the landing page should map directly to the rules, just in simpler
language with examples. If the goal and the examples live in the rules themselves, the page is a direct
projection of them — and that is the most transparent thing it can be.*

## Context

A meta-project advertises its benefits poorly: the rules are abstract, so the value (felt across real
projects) doesn't show. The fix is **not** a separate marketing page — that would be a second copy that
can drift from the truth (the exact failure the invariants forbid). The fix is to **bake the goal and
the examples into the rules**, and render the page *from* them.

An audit found the existing "every invariant carries an example" acceptance criterion is **aspirational,
not met**: concrete examples cluster in B1; most invariants lean on framework name-drops or have none;
none carry a contrastive *with/without* pair.

## Decision

1. **Bake goals + examples into the rules.** Each invariant's `signature-catalog` entry gains:
   - **`why`** — the goal / benefit in one line, written **agents-first, humans-second** (clarity of
     intent for the agent that reads it; legibility of value for the human).
   - **`honored` / `violated`** — a contrastive pair: what a project looks like *with* the invariant
     vs. the failure *without* it. *(e.g. `inv-clarify-before-commit` — honored: the agent asks about a
     vague requirement before coding; violated: it silently picks a reading, builds it, it's wrong.)*

2. **The meta-rule (make the requirement real).** Every invariant MUST carry `why` + `honored` +
   `violated`; **a change that adds or edits an invariant without updating its examples is a
   conformance failure** (extends `spec-0002` §4 catalog-coverage). Presence is the enforceable floor;
   *did-a-change-action-an-example-it-failed-to-update* is the substantive check (weakly checkable, like
   SI-1). This is Trellis's iron rule + referential integrity applied to its own rule-set.

3. **The landing / benefits page is a *derived, simplified view* of the ratified rules.** It is
   generated from the catalog's `why` / `honored` / `violated`. **Hard constraint: no claim on the page
   without a rule behind it.** The page cannot drift from the truth because it *is* the truth, made
   readable — Trellis applying its own **transparency floor (D1)** and **no-silent-drift** to its own
   marketing.

4. **"No silent drift" is a recorded cross-cutting theme, not a new invariant.** An invariant review
   confirms it is **D1 (transparency) generalized across every kind of drift** — directional-flow +
   graph-maintenance (artifacts), B8 (frameworks), self-improvement (deliberate evolution, not drift) —
   the floor that makes all of it "never *silent*." *Drift is allowed, but never silent.* Recorded in
   the invariant-set framing; **not** a new rule (minimal-first).

## Consequences

- **`signature-catalog` schema extended** (`spec-0002`): `why` / `honored` / `violated` per entry;
  the conformance check learns the completeness + sync rule.
- **The benefits page + the landing "how it works" section derive from the catalog** — a `docs/`
  resource + the landing page, both generated (by hand first; a render step later).
- **Named preset profiles** — the two distinct postures **A · conductor** (adopt one framework,
  strictly) and **B · author-adapt** (synthesize/evolve) ship as shelf expression profiles; **C ·
  grow-from-scratch is *the seed*, a minimal author starting point that grows — not a third preset**
  (its distinctness is a trajectory + starting content, not a dial config). These map to
  `decision-0009`'s three validation scenarios and seed the advisor-CLI's **A / B / C-seed / Custom**
  onboarding. *(Companion artifact; recorded here as the page's "three ways to use Trellis.")*
- **Folds with** the owed un-merge revision (`decision-0018`) + the catalog's own ratification —
  one careful pass at the invariants/catalog.

## Open questions

- **Render mechanism** — the page is generated by hand at first; when does a real render step (catalog
  → page) get built, so the "no claim without a rule" constraint is *mechanically* enforced, not just
  by discipline?
- **Preset dial precision** — the exact per-invariant `C1`/`C2` for A and B (this decision fixes that
  they exist + their postures; the numbers land with the preset artifact).
- **`why` audience** — one line serving agents-first *and* humans, or two registers? (Start with one.)
