Governance for agentic development

The structure your agents grow along.

Trellis is a governance layer for agentic software development. It fits whatever methodology your project already uses, teaches it to your coding agents, and enforces a small set of invariants — so a process glitch never has to happen twice.

install, then set up
$ curl -fsSL https://raw.githubusercontent.com/gundisalwa/trellis/main/install.sh | sh
$ trellis setup # then set it up in your project

The installer just drops a single binary — it doesn't run anything on its own; you run trellis setup. Remove it any time: trellis remove inside a project, or trellis uninstall for the binary.

On Claude Code? Skip the binary — add it as a plugin (covers the overlay natively):

/plugin marketplace add gundisalwa/trellis
/plugin install trellis@trellis
/trellis:setup

The problem

Agents move fast. Without structure, they lose the thread.

Trellis holds the load-bearing rules so your agents can move quickly without building on shifting ground.

01

Referential integrity

Every artifact — research, decisions, specs, code — points to the settled ground it depends on. Agents build on ratified truth, never a draft that's still moving.

02

Knowledge flows back

When a downstream discovery contradicts an upstream doc, the doc gets updated — not just the code. Learnings propagate instead of forking.

03

A glitch, once

Friction becomes a rule where it fires. The same process failure doesn't recur every few weeks because nothing captured the lesson.

With vs without

The same project, guarded.

Without Trellis

  • An agent codes against a spec that's still being edited; the spec shifts and the work is built on a version that no longer exists.
  • A human-gated decision gets merged with no approval — silently.
  • The same flaky step fails every week and everyone just re-runs it.

With Trellis

  • Implementation reads only ratified specs; downstream never consumes a draft.
  • A human-gated handover performed without its approval is surfaced, loudly.
  • The recurring failure becomes a checkable rule that rides the PR you already write.

How it works

One command. It reads your project, you choose the fit.

Trellis rides your existing harness (Claude Code today). It detects, proposes, and — only with your go-ahead — composes itself onto your project. No runtime, no lock-in.

01 · detect

Find the harness

Rides the agent CLI your project already uses.

02 · profile

Pick a posture

03 · mode

Alongside or rewrite

Overlay next to your rules, or morph them in on a branch.

04 · apply

You approve

Augment-never-clobber. Trellis proposes; the merge is yours.

The core

A small set of invariants, expressed at your strength.

Not a process — the layer above it. A handful of load-bearing invariants (directional flow, ratifiable artifacts, gate-at-handover, independent judgment, transparency…), each set along two dials: how strictly it's enforced, and who gates it. Everything else, Trellis respects.

It grounds out in real artifacts

Trellis never just describes process. It produces and enforces concrete, project-specific artifacts — a real instructions file, real gates, a real conformance check. If it can't check it, it doesn't claim it.

It fits, it doesn't dictate

Gatekeepers are whatever your project already declares — detected and respected, not imposed. Trellis enforces the invariants and gets out of the way of your methodology.
